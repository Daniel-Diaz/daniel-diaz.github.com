liveData :: IntMap (IntMap LiveValue)

IntMap of live data. The first key is the PID of an user. If a PID is not present, there is no live data of that user. You should look at the AwayServer. The second key (key of the inner map) refers to the initial time of the data in a binary store. Depending on the usage of the data, initial times will vary.

Live value maps should be manipulated strictly, while live values should be manipulated lazily.

awayServer :: AwayServer

Server to send the old data.

type DecodedData = Decoded (Maybe Double)

Decoded data with holes.

type Counter = Int

The counter specifies the age of some live data. Live data is data kept in memory for quick access.

The counter has two limits:

  • First limit: After this limit is reached, live data is dropped from the global state, but the counter keeps growing. Therefore, the counter will stay in memory.
  • Second limit: After this limit, the file pointed by the Key is sent to S3. The counter is removed from memory as well.

data LiveValue

A live value in memory. It might or might not contain decoded data (lvData). If it does not contain the decoded data, it can be read from the local disk. The counter (lvCounter) tells you how much time this value has been on memory. See Counter.

Request handlers

getTransferR

Arguments

:: Int

PID

-> Int

Initial time

-> Handler TypedContent 

Send a chunk of data to the requester. It sends a 404 response if the chunk can't be found. If not found in the server, it will ask to the configured AwayServer. The 404 response contains an error description in its body. Otherwise (200 response), the body contains the chunk in binary format.

When the transference is accomplished, the response might be delayed by some seconds. This is because the server must make sure that - in case that the chunk was retrieved from the away server - the chunk no longer stays stored at the away server, thus avoiding conflicting copies to roll around. If, after a fixed number of tries, the file from the away server is not deleted, this server will stop trying and will report an error. However, even if an error is reported, the transferred data will still be included in the response body. A 200 response means that everything worked as expected. A 500 response means that the deletion on the away server failed. The details will be prompted in the server, and the HTTP response will contain the error in a X-Error header.

If a transference is done from an away server, but the data transferred was found corrupt, a 501 response will be sent, with the response body describing the error.
